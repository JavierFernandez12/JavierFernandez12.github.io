Web Deployment Challenges

Challenge #1: PHP Dice Game

PHP / Logic / Forms
1. Challenge Overview

Following the video guidelines, I developed a dynamic web application using PHP that simulates a dice game against a virtual opponent. The project consists of two pages: 1. Input Page (Form): The user selects the number of dice to roll, the number of faces per die, and sets the opponent's target score. 2. Results Page (Logic): The server processes the input, generates random rolls, displays the corresponding dice images, calculates the total points, and determines if the player won or lost against the set score.

2. Key Complex Concepts Learned
  • Server-Side Form Handling ($_POST): I learned how to capture user input securely from an HTML form using the PHP superglobal $_POST and validate that data before processing.
  • Dynamic Asset Rendering: Using loops (for) combined with random number generation (rand()) to dynamically construct the HTML <img> tags, ensuring the correct die image matches the random number generated.
3. Personal Reflection

It was very satisfying to see the logic come to life. At first, I had some trouble with the image paths, ensuring that if the random number was '3', the script loaded 'dice_3.png'. Once fixed, seeing the "You Win!" message appear based on my own code logic felt great.

4. Proposed Improvements

Currently, the game restarts from zero. To improve it, I would use PHP Sessions to remember the user's settings (number of dice/faces) so they don't have to re-enter them every time they want to play again. I would also add CSS animations to make the dice "roll" on the screen.

5. Technical Glossary
  • PHP (Hypertext Preprocessor): A server-side scripting language designed for web development.
  • $_POST: A PHP super global variable used to collect form data sent with the HTTP POST method.
  • Control Structures: Blocks of code (if/else, loops) that determine the flow of execution based on conditions (e.g., if total > opponent).
  • Rand(): A built-in PHP function used to generate a random integer.
  • Backend Logic: The part of the application that runs on the server (calculations, database interactions) rather than the browser.
6. Assignment Rating & Feedback
Rating: 8.5/10
Reason: A classic but effective exercise to understand the request-response cycle in web apps. Improvement for the statement: The video was clear, but providing the image assets (dice icons) beforehand would save time searching for them online.

Challenge #2: PHP & MySQL Games Management System

PHP / MySQL / CRUD
1. Challenge Overview

I created a relational database named `BD_GAMES` containing three main tables: `GAMES`, `PLAYERS`, and `PARTIDAS` (Matches). The goal was to build a PHP web application with a dashboard (`index.php`) providing direct links to manage these tables and displaying a control summary (e.g., total number of games, players, matches, and advanced statistics like players participating in 2, 3, or more than 3 matches). Each table management section required dedicated pages to view the data in HTML tables, insert new records, modify specific fields by ID, and delete records. The final code was submitted via Git.

2. Key Complex Concepts Learned
  • Relational Data Integrity: I successfully mapped Foreign Keys, ensuring that `ID_GAME` in the matches table referenced the `GAMES` table, while `ID_PLAYER1` and `ID_PLAYER2` correctly linked to the `PLAYERS` table.
  • CRUD Architecture: Structuring a PHP application to handle Create, Read, Update, and Delete operations dynamically across multiple interconnected tables.
3. Personal Reflection

Building the summary control board with advanced SQL queries was challenging. Recording the final demonstration video highlighting all functionalities in under 1:01 minutes forced me to ensure the UI flow was smooth and error-free.

4. Proposed Improvements

I would upgrade the application by using PDO (PHP Data Objects) and prepared statements for all database interactions to ensure robust protection against SQL injection attacks during the insert and modify processes.

5. Technical Glossary
  • CRUD: Create, Read, Update, Delete - the four basic operations of persistent storage.
  • Foreign Key (FK): A column that creates a link between two tables, ensuring referential integrity.
  • Git: A version control system used to track changes in the source code, which was a requirement for the final delivery.
6. Assignment Rating & Feedback
Rating: 8/10
Reason: A highly practical exercise for mastering database interactions from a web frontend.

Challenge #3: Advanced Redis Caching with PHP

PHP / Redis / Performance
1. Challenge Overview

For the "App Aceitunas" project, I built a PHP web app to manage Harvesters (`Vareadores`) and Olive Trees (`Olivos`) using a Many-to-Many relational structure, allowing users to view, add, modify, and delete records. The key technical challenge was implementing Redis as an intermediate caching layer to optimize MySQL queries. I configured specific expiration times (TTL) for the cache: 10 seconds for numeric keys and 60 seconds for text-based keys. The system architecture follows a strict flow: check Redis first; if the key is missing, fetch the data from MySQL, return it, and store it in Redis for future requests.

2. Key Complex Concepts Learned
  • Server Administration & Compilation: I learned how to install the `php-redis` extension from source via Git, which involved using `phpize` and fixing critical C11 atomic header incompatibilities (`zend_atomic.h`) within the XAMPP environment using stream editors (`sed`).
  • Redis Data Structures in PHP: I managed data beyond simple strings by utilizing Redis Hashes (`hSet`, `hGet`, `hGetAll`) to store user objects and Lists (`lPush`, `lRange`, `rPop`) to manage task queues directly through the PHP Redis object.
3. Personal Reflection

The initial setup was demanding, especially troubleshooting the header compilation errors in Linux. However, successfully patching the system and seeing the `phpinfo()` screen confirm the Redis module was active felt incredibly rewarding. Understanding the logic of the caching flow diagram changed how I view web performance optimization.

4. Proposed Improvements

Instead of relying solely on TTL for cache expiration, I would implement active cache invalidation. This means the system would proactively delete or update the Redis cache whenever a Harvester or Olive Tree record is modified or deleted in the MySQL database.

5. Technical Glossary
  • Cache: A hardware or software component that stores data so that future requests for that data can be served faster.
  • TTL (Time-to-Live): A mechanism that limits the lifespan or lifetime of data in a computer or network, used here to expire old cache data.
  • phpize: A shell script used to prepare the build environment for a PHP extension.
  • build-essential / autoconf: Packages containing compilers and necessary tools for building software from source code in Linux.
6. Assignment Rating & Feedback
Rating: 9.5/10
Reason: A challenging but extremely valuable introduction to enterprise-level performance optimization and server administration.
