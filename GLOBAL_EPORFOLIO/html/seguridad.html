Challenge #1: Kali Purple Attack Simulations

Hydra / Pydictor / DVWA
1. Challenge Overview

This comprehensive challenge focused on simulating brute-force attacks using Kali Purple. The work was divided into three stages: (1) Preparation: Creating custom wordlists using Pydictor and merging them with Dymerge. (2) Infrastructure Attack: Setting up an SSH server and using Hydra to crack a user's password. (3) Web Attack: Deploying DVWA and using Hydra to brute-force a login web form via HTTP POST requests.

2. Key Complex Concepts Learned
  • Protocol-Specific Brute Force: I learned the significant difference between attacking a standard network protocol like SSH versus a stateless web protocol like HTTP, where you must identify failure strings.
  • Dictionary Engineering: Understanding that effective hacking isn't just about the tool, but about the quality of the "ammunition" (wordlists).
3. Personal Reflection

The experience was a roller coaster. The SSH part was exciting—seeing the password crack successfully gave me a rush. However, the HTTP part caused some frustration because getting the correct syntax for Hydra's http-post-form was tricky and required trial and error.

4. Proposed Improvements

To improve the workflow, I would write a Bash script to automate the dictionary generation. For the web attack, I would use Burp Suite to capture the request headers instead of browser tools for better accuracy.

5. Technical Glossary
  • Brute-force: A cryptanalytic attack that consists of trying every possible combination of characters.
  • Hydra: A fast and flexible network login cracker.
  • False Positive: A result which wrongly indicates that a particular condition is present.
  • HTTP POST: The HTTP method used to send data to a server (login forms).
  • Wordlist: A collection of words or strings used for dictionary attacks.
6. Assignment Rating: 9/10
Feedback: Excellent practice covering the full attack cycle.

Challenge #2: Advanced Permissions & Forensics

Linux / ACLs / Bash
1. Challenge Overview

This practice focused on hardening the filesystem and mastering command-line auditing. Phase 1: I managed a multi-user environment (`monon1`, `tronko2`, `birmingan3`) implementing advanced permissions like SGID for collaborative folders, Sticky Bit for secure temporary directories, and ACLs for granular access. Phase 2: I performed forensic-style searches using grep, find, and locate to identify critical errors in logs and hidden configuration files within a complex directory structure.

2. Key Complex Concepts Learned
  • Special Permission Bits (SGID & Sticky): Understanding how SGID forces new files to inherit the parent directory's group (essential for team collaboration) and how Sticky Bit prevents users from deleting each other's files in shared folders like /tmp.
  • Access Control Lists (ACLs): I learned that standard `rwx` permissions are limited. ACLs allow granting specific permissions to a single user without changing the group ownership.
3. Personal Reflection

Calculating the umask (subtracting from 777) was confusing at first. However, verifying that user `birmingan3` could not delete `monon1`'s file due to the Sticky Bit gave me a strong sense of control and security. The search exercises with regex made me feel very efficient.

4. Proposed Improvements

I would create a setup script that automatically creates the users and directory structure to reset the environment quickly for testing. Also, I would practice using chattr (immutable bit) for even higher security on critical logs.

5. Technical Glossary
  • SGID (Set Group ID): A special permission bit that forces files created inside a directory to inherit the directory's group ownership.
  • Sticky Bit: A permission bit set on directories that restricts file deletion to the file's owner only (used in `/tmp`).
  • ACL (Access Control List): A list of permissions attached to an object giving specific access rights to users or groups.
  • Umask: A command that determines the default file permissions for newly created files.
  • Grep: A command-line utility for searching plain-text data sets for lines that match a regular expression.
6. Assignment Rating: 8/10
Reason: Very useful for real-world sysadmin tasks.

Challenge #3: Malware Research Presentation

Malware / Teamwork / Git
1. Challenge Overview

As part of "Team 1", we researched and delivered a 10-minute oral presentation covering Viruses, Macro Malware, Rootkits, and Drive-by downloads. The strict requirements included speaking without notes, covering attack techniques, consequences, and analyzing a real-world case study. Additionally, we practiced version control by committing our progress after every class ([clase]) and home session ([casa]).

2. Key Complex Concepts Learned
  • Rootkit Persistence mechanisms: Understanding how rootkits operate at the kernel level (Ring 0) to hook system calls and hide their presence from antivirus software.
  • Drive-by Download Architecture: Learning how attackers inject malicious scripts into legitimate websites to exploit browser vulnerabilities and install malware silently.
3. Personal Reflection

Speaking in public without notes was nerve-wracking at first, but preparing the script thoroughly gave me confidence. Investigating real cases of "Drive-by downloads" made me feel much more cautious about web browsing.

4. Proposed Improvements

Our slides were text-heavy. If I could do it again, I would use more diagrams to explain the Rootkit architecture and maybe include a safe, recorded demo of a Macro execution.

5. Technical Glossary
  • Rootkit: A set of software tools that enable an unauthorized user to gain control of a computer system without being detected.
  • Macro Virus: A virus written in a macro language (like VBA) that is embedded in documents and runs when the document is opened.
  • Drive-by Download: The unintentional download of malicious code to a computer or mobile device that leaves a user open to a cyberattack.
  • Kernel Hooking: A technique used by rootkits to intercept function calls between software and the operating system.
  • Zero-day Exploit: A cyberattack that occurs on the same day a weakness is discovered in software.
6. Assignment Rating: 8.5/10
Reason: Great practice for soft skills and Git discipline.

Challenge #4: Web App Security (SQLi & XSS)

Python Flask / PHP / OWASP
1. Challenge Overview

This lab focused on exploiting and patching two of the most critical web vulnerabilities: SQL Injection (SQLi) and Cross-Site Scripting (XSS). I developed vulnerable web apps using Python Flask and PHP. First, I bypassed a login screen using SQL payloads like ' OR 1=1 --. Then, I injected JavaScript alerts using Reflected and Persistent XSS. Finally, I secured the code using Prepared Statements and Input Sanitization (htmlspecialchars).

2. Key Complex Concepts Learned
  • SQL Injection (String Concatenation vs. Prepared Statements): I understood that concatenating user input directly into SQL queries allows attackers to manipulate the logic of the database. The fix is treating input as parameters, not executable code.
  • XSS Types (Reflected vs. Persistent): Reflected XSS happens immediately via URL parameters (e.g., search bars), while Persistent XSS stores the malicious script in the database, executing it every time a victim loads the page (e.g., comments section).
3. Personal Reflection

It was shockingly easy to bypass the login with a simple string. It made me realize how fragile legacy code can be. Fixing the vulnerabilities gave me a sense of responsibility; writing secure code is not optional, it's mandatory.

4. Proposed Improvements

I manually tested the payloads. To improve, I would use automated tools like SQLMap to see how deep the exploitation can go, and maybe implement a WAF (Web Application Firewall) (like ModSecurity) to see how it filters these attacks.

5. Technical Glossary
  • SQL Injection (SQLi): A code injection technique used to attack data-driven applications by inserting malicious SQL statements into entry fields.
  • Cross-Site Scripting (XSS): A vulnerability that enables attackers to inject client-side scripts into web pages viewed by other users.
  • Sanitization: The process of cleaning input data to prevent it from being interpreted as code (e.g., converting < to &lt;).
  • Prepared Statement: A feature used to execute the same (or similar) SQL statements repeatedly with high efficiency and security against SQL injection.
  • Payload: The part of the malware or exploit code that performs the malicious action.
6. Assignment Rating: 9.5/10
Reason: Very practical. Seeing the code break and then fixing it is the best way to learn secure coding. Improvement for the statement: It would be interesting to include a "Blind SQL Injection" challenge, where you don't see errors but have to guess the database structure.

Challenge #5: DMZ & Network Security Architecture

pfSense / Squid Proxy / OpenVPN
1. Challenge Overview

Working in a team of three, we researched and documented core network security concepts: Firewalls, DMZ, Proxy, and VPN. We then designed and implemented a virtualized network architecture across two challenges. The final environment included two pfSense firewalls (External and Internal), an OpenVPN server located in the DMZ, a dedicated Squid Proxy server on the LAN, and an internal client machine. We concluded the project with an oral presentation.

2. Key Complex Concepts Learned
  • Advanced Firewall Routing & NAT: I configured the External pfSense to handle NAT for internet access and created strict rules to block direct traffic from the DMZ to the LAN, only allowing specific exceptions like the VPN accessing the Proxy on port 3128.
  • Proxy Filtering & ACLs: I learned how to configure Squid to restrict network access. I implemented Access Control Lists (ACLs) to block specific domains (like Facebook and YouTube) using a blacklist file and enforced time-based restrictions to only allow browsing during working hours.
3. Personal Reflection

Connecting the different subnets (WAN, DMZ, LAN) correctly in VirtualBox was a great puzzle. Testing the connectivity was incredibly rewarding; watching the `/var/log/squid/access.log` actively register `TCP_DENIED` when I tried to access a blocked site confirmed that my security policies were actually working.

4. Proposed Improvements

Instead of setting up all the virtual machines manually, I would like to explore Infrastructure as Code (IaC) tools like Vagrant to automate the deployment of the network layout and the pfSense configurations.

5. Technical Glossary
  • DMZ (Demilitarized Zone): A perimeter network that protects an organization's internal local-area network from untrusted traffic.
  • pfSense: An open-source firewall and router software distribution based on FreeBSD.
  • Squid: A caching and forwarding HTTP web proxy.
  • ACL (Access Control List): Rules configured in the proxy or firewall to permit or deny network traffic.
  • OpenVPN: A virtual private network system that implements techniques to create secure point-to-point or site-to-site connections.
6. Assignment Rating: 9/10
Reason: A robust, enterprise-level simulation that tied many networking and security concepts together perfectly.

Challenge #6: OSINT & Phishing Toolkit Analysis

OSINT / Malware Analysis / Footprinting
1. Challenge Overview

This assignment required conducting a mini-study on various cybersecurity tools, where I had to install, test, and document an example of use for each tool. The scope covered Phishing Toolkits (analyzing headers, IP reputation, links, and attachments) and Open-Source Intelligence (OSINT) tools for investigating passwords, social engineering vectors, emails, social networks, and utilizing specialized search engines. The findings were presented using a textless PowerPoint and Markdown documentation.

2. Key Complex Concepts Learned
  • Malware Attachment Analysis: I learned how to safely analyze suspicious files and links using sandboxing and cloud analysis tools like VirusTotal and Any.run.
  • Footprinting and OSINT: Understanding how to gather publicly available data. I used specialized engines like Shodan and checked credential leaks utilizing databases like HaveIBeenPwned.
3. Personal Reflection

Discovering the sheer volume of information exposed on the internet through tools like Shodan or simple Google Dorks was eye-opening and slightly alarming. Presenting the capabilities of these tools to the class without relying on text slides forced me to truly understand how they functioned rather than just reading definitions.

4. Proposed Improvements

The toolkit document also contained advanced sections for Red Team operations (like Privilege Escalation and Command & Control) and Blue Team incident management. I would love to extend this study to practically test tools like Cobalt Strike or Wazuh in a lab environment.

5. Technical Glossary
  • OSINT (Open-Source Intelligence): Data collected from publicly available sources to be used in an intelligence context.
  • Phishing: The fraudulent practice of sending emails purporting to be from reputable companies to induce individuals to reveal personal information.
  • Shodan: A search engine designed to locate specific types of computers, webcams, routers, and servers connected to the internet.
  • Sandboxing: A security mechanism for separating running programs, often used to execute untested or untrusted programs (e.g., Any.run).
6. Assignment Rating: 8.5/10
Reason: Excellent exposure to the actual tools used daily by security professionals for reconnaissance and defense.
